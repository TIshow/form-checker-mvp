# analysis — サーブ解析とフィードバック生成

3D関節データからバイオメカ指標を算出し、改善点を言葉で返す。
**純 numpy のみに依存**。3D復元 (GVHMR) の環境は不要で、通常の Python で動く。

## 構成

変更頻度の違う3つの層に分けている。

| モジュール | 役割 | 変更頻度 |
|---|---|---|
| `serve.py` | 計測（幾何・運動学・指標） | 低。物理計算は変わらない |
| `feedback.py` | 判定ルールと閾値 | **高**。データが貯まるたび調整する |
| `report.py` | 表示 | 将来 Web UI に置換される |

P1のパイプラインは `serve.py` だけを import すればよく、閾値の調整が
物理計算に影響しない。

## 使い方

```python
import analysis

metrics, feedback = analysis.analyze_from_files(
    "gv_joints.npy", "gv_com.npy", "gv_upaxis.npy", fps=120.0)
print(analysis.format_report(metrics, feedback))
```

CLI:

```bash
python -m analysis --joints gv_joints.npy --com gv_com.npy \
                   --upaxis gv_upaxis.npy --fps 120
```

Colab で使う場合は、このディレクトリを Google ドライブに置いて
`sys.path` に追加する（`notebooks/README.md` 参照）。

## 閾値の信頼度を3段階に分ける

閾値の正しさがそのままフィードバックの正しさになる。「何を根拠にその数字を
決めたか」をコード上で区別し、根拠のないものでは断定しない。

| Tier | 根拠 | 扱い |
|---|---|---|
| **A** | 力学的原理。反例が考えにくい | 断定してよい |
| **B** | 文献にプロの実測レンジがある | 参考として提示する |
| **C** | 根拠なし | **判定しない**。数値を表示するだけ |

### なぜこの区別を入れたか

初期実装は体幹の傾きが35°を超えると「傾きすぎです、軸を保ちましょう」と
警告していた。根拠として挙げた「プロは水平から48°傾いている」という値は、論文本文を
確認したところ**存在しなかった**。検索結果の要約から孫引きしたものだった。

閾値の大半は「それらしい」と思った値を置いただけであり、出典として挙げた
論文にその数字があるかを照合していなかった。**出典を書くこと自体が、
実際には無い裏付けがあるように見せる**という、より悪い状態を作っていた。

### 現在の分類

| 指標 | Tier | 根拠 |
|---|---|---|
| キネティックチェーンの順序 | A | 中心から末端へ順に加速するのが効率的 |
| 打点タイミング（重心頂点との差） | A | 落下開始後に打つとエネルギーが逃げる |
| 接球時の肘の伸展 | B | プロは屈曲 30.1±15.9°（関節角 約150°） |
| 沈み込みの膝角 | B | プロのトロフィー時 屈曲 64.5±9.7°（関節角 約115°） |
| 体幹の傾き(接球時) | C | 文献はトロフィー時25.0±7.1°のみ報告。**接球時の基準値が無い**ため判定しない |
| 打点/頭の高さ比 | C | 標準指標ではなく比較対象がない |
| X-factor | C | 研究の大半がゴルフ。サーブでの適正レンジ未確認 |

膝屈曲は個別研究で **45.0°〜82.8°** と幅が非常に大きい。Tier B は
「プロと違うから直せ」ではなく参考情報として提示する。

**角度の定義が一致している保証はない。** 論文がどの基準面・どの体節ベクトルで
測ったかまでは確認できていない。定義が違えば数値を直接比べる意味がないため、
Tier B でも逸脱をもって欠点と断じない。

### 出典の扱い

Tier B に使えるのは、**論文本文を開いてセクション・図番号まで確認した値だけ**。
検索結果の要約を根拠に「文献値」と称してはいけない。

[1] Kinematics characteristics of key point of interest during tennis serve
    among tennis players: a systematic review and meta-analysis.
    Frontiers in Sports and Active Living (2024).
    https://doi.org/10.3389/fspor.2024.1432030

| 使用値 | 確認箇所 |
|---|---|
| 前脚の膝屈曲(トロフィー時) 64.5 ± 9.7° | [1] Section 3.4, Figure 3 |
| 肘屈曲(接球時) 30.1 ± 15.9° | [1] Section 3.6, Figure 7 |
| 体幹傾斜(トロフィー時) 25.0 ± 7.1° | [1] Section 3.4, Figure 2 |

## 設計方針: プロ比較をしない

「プロと比べて差が大きい所を指摘する」は破綻する。プロ同士でもフォームは大きく異なり、
差は必ずしも欠点ではない。体格が違えば同じ関節角度は再現できないし、する必要もない。

代わりに**誰にでも当てはまる力学的原理**で判定する。参照動画も権利処理も不要で、
かつ最も価値の高い指摘が得られる。

| 判定 | 根拠 |
|---|---|
| キネティックチェーンの順序 | 体の中心から末端へ順に加速するのが力学的に効率的 |
| 打点タイミング | 重心が落ち始めてから打つと力が逃げる |

量的指標（膝・肘）は Tier B として文献値を参考表示するに留める。
打点の高さ比・捻転差は Tier C で判定しない。

## フィードバックの作法

- **一度に1〜2件しか出さない**。人は複数の修正キューを同時に処理できず、
  15件の指摘は0件と同じになる
- 優先度は「原理違反 > 大きな逸脱 > 小さな逸脱」
- 判定はルールで決定論的に行う。将来 LLM を使う場合も**文言の生成のみ**に限定する。
  判定を任せると数値を誤解して嘘の助言を作る

## 測定できないものは主張しない

キネティックチェーンの連鎖は隣接体節の時間差が 20〜40ms しかない。
30fps では 1フレーム=33ms あり**分解できない**ため、`CHAIN_MIN_FPS`(60fps) 未満では
順序を判定せず、レポートにその旨を明示する。

実際、この保護が無かった頃は 30fps の実データで「1フレーム差」を根拠に
誤った指摘を出していた。順序を評価したい場合は 120/240fps で撮影すること。

## テスト

```bash
uv pip install -e ".[dev]"
pytest
```

`tests/synth.py` が合成サーブを生成する。タイミングは秒で定義してあり、
fps を変えても同じ実時間の動作になるため、フレームレート依存の挙動を検証できる。

このテストは実際に2つのバグを検出した。閾値やルールを触ったら必ず走らせること。

## 既知の限界

- **閾値が暫定**。順序・タイミング系は理論から断定できるが、
  「膝を何度まで曲げるべきか」等は文献値かデータ蓄積で確定させる必要がある
- **コート座標が未定義**。前後左右が分からないため踏み込み方向を評価できない
- **1本のサーブのみ**を前提。複数本の比較・経時変化の追跡は未実装
