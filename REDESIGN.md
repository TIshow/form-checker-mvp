# テニスフォーム分析 — 設計方針

**状態: P0（技術検証）完了。P1（パイプライン化）が次。**
決定済みの事項と、その理由を記録する文書。実装は `analysis/` と `notebooks/` にある。

---

## 1. 実現したいこと

> 動画を撮って「なんか崩れてたね」で終わらせず、
> **データに裏付けられた、行動に移せる学び**を返すツール。

抽出したい対象:

- **重心 (Center of Mass)** と体重移動
- **キネティックチェーン**（腰→体幹→肩→肘→手首の力の伝達順序と効率）
- **関節角度**（肘、肩の外旋、膝、股関節）
- **打点**（高さ・体との位置関係・タイミング）
- サーブ固有: トロフィーポーズ、ラケットドロップ、脚のドライブ、体幹の回転

## 2. なぜ2Dを捨てたか

初代MVPは MediaPipe の2D姿勢推定で作られていた。

| 項目 | 2Dでの問題 |
|---|---|
| 重心 | ピクセルの加重平均。単位も床基準もなく**物理的に無意味** |
| 体重移動 | 奥行きが無く**原理的に測れない** |
| 関節角度 | カメラアングルで見かけの角度が歪む |
| 打点 | ボール検出がモック（偽データ）だった |
| プロ比較 | 時間同期も体格正規化もないフレーム単位差分は同じ局面を比べていない |

2Dの上に何を積んでも到達できないため、土台を3Dに置き換えた。
旧実装は `legacy-2d` ブランチに保存してある。

### 2Dの現在の役割

2Dが劣っているのではなく、**測れるものが違う**。実際 GVHMR は内部で
YOLO(人物検出) と ViTPose(2D姿勢) を使っており、2Dは3Dの材料になっている。

| 2Dが担う | 2Dに任せない |
|---|---|
| 3D復元の入力、ボール・ラケット検出 | 重心・体重移動 |
| 前処理のゲート（GPUを使う前に解析可能か判定＝コスト削減） | 関節角度 |
| 検証・可視化のオーバーレイ、即時プレビュー | キネティックチェーン |

**ユーザーに「2Dモード」を見せない。** 意味のない数字を信じさせることになり、
旧アプリの問題が再発する。2Dは裏方に徹する。

## 3. パイプライン

```
動画
 └─▶ ① 3D人体復元 (world-grounded SMPL)      ← notebooks/ (GPU必須)
     └─▶ ② バイオメカ量の算出 (重心・関節角)    ← analysis/serve.py
         └─▶ ③ 動作の意味づけ (フェーズ・連鎖・打点)
             └─▶ ④ フィードバック生成          ← analysis/feedback.py
```

### ① 3D人体復元 — GVHMR を採用

**GVHMR** (SIGGRAPH Asia 2024 / TPAMI 2026) を採用。重力方向を推定する
Gravity-View座標系を持ち、**接地・傾き・バランスが安定して取れる**ため、
重心分析に本質的に効く。

代替として WHAM (CVPR 2024) と TRAM (ECCV 2024) を検討したが、TRAM は
軌跡の数値精度は高いものの物理的な破綻が出やすくバランス評価に不向き。

実装上の注意（P0で判明）:

- 静止カメラなら `-s` フラグで SLAM(DPVO) を回避でき、導入が大幅に楽になる
- **Python 3.10 が必須**。3.12 では `numpy==1.23.5` がビルド不可、`chumpy` が動かない
- 出力の `smpl_params_global` は名前に反して **SMPL-X パラメータ**（`body_pose` が63）。
  SMPL として扱うと壊れる

### ② バイオメカ量の算出

SMPL の24関節に**人体計測学的な体節質量比(De Leva)**を割り当て、
world座標で全身重心を計算する。実装は `analysis/serve.py`。

現在算出しているもの: 重心の高さ・軌跡、膝屈曲、肘角度、体幹の傾き、
X-factor(肩-腰の捻転差)、各体節の角速度。

発展: OpenCap / OpenSim 相当の筋骨格モデルへ変換し臨床レベルの関節角・トルクへ。

### ③ 動作の意味づけ

- **フェーズ分割** — 現在は「重心の谷＝沈み込み」「ラケット側手首の最高点＝打点」
  で検出。P0でこれが実際の接球フレームと一致することを確認済み。
  1本のクリップに複数動作が混在する場合の切り出しは未実装
- **キネティックチェーン** — 各体節の角速度ピークの時刻順序
- **打点検出** — 現在は手首の高さのみ。ボール/ラケット検出（YOLOファインチューン）と
  **打球音**による接触検出は未実装

### ④ フィードバック生成 — プロ比較は採らない

**当初の「プロと比較して差分を指摘する」方針は破棄した。** 理由:

- プロ同士でもフォームは大きく異なり、**差は必ずしも欠点ではない**
- 体格・柔軟性が違えば同じ関節角度は再現できないし、する必要もない
- どの差が重要かを決める根拠がない

代わりに**誰にでも当てはまる力学的原理**で判定する。参照動画も権利処理も不要で、
かつ最も価値の高い指摘が得られる。

フィードバックの源泉を実用順に並べると:

| # | 源泉 | プロ動画 | 状態 |
|---|---|---|---|
| ① | **力学的原理**（連鎖の順序、打点タイミング） | 不要 | **実装済み** |
| ② | 自分の過去との比較（経時変化） | 不要 | 未実装 |
| ③ | 自分の良い球 vs 悪い球 | 不要 | 未実装 |
| ④ | プロとの比較（体格正規化＋DTW整列） | 必要 | 将来 |

①②③はプロ動画が要らず、しかも①③は個人に最適化された助言になる。
プロ比較は最後でよい。

作法:

- **一度に1〜2件しか出さない**。人は複数の修正キューを同時に処理できず、
  15件の指摘は0件と同じ
- 判定はルールで決定論的に行う。LLM を使う場合も**文言の生成のみ**に限定する。
  判定を任せると数値を誤解して嘘の助言を作る

## 4. 測定できないものは主張しない

P0の実データで、システムが**1フレーム差を根拠にキネティックチェーンの
順序の逆転を指摘した**。30fps では1フレーム=33ms、一方で連鎖の隣接体節の
時間差は 20〜40ms しかない。**そもそも分解できない量について主張していた**。

対処として `CHAIN_MIN_FPS`(60fps) 未満では順序を判定せず、レポートに
その旨を明示するようにした。

**撮影要件: キネティックチェーンを評価するなら 120fps 以上**
（iPhoneのスローモード等）。この制約は今後の指標追加にも効いてくる。

数値が出ることと、その数値が意味を持つことは別である。

## 5. システム構成

```
[スマホ / ウェブ]  動画を撮ってアップロード
        ▼
[ストレージ + API]  ジョブ登録（非同期）
        ▼
[サーバーレスGPU]   3D復元 → 打点検出 → 指標算出 → フィードバック生成
        │           重い処理はここだけ。使った分だけ課金・アイドル費用ゼロ
        ▼
[結果DB]           3Dモーション・指標・履歴
        ▼
[アプリ]           3Dビューア + フィードバック + 進捗グラフ
```

**GPUをローカルに買う必要はない。** サーバーレスGPU（Modal / Replicate / fal.ai 等）
なら動画1本あたり数円〜十数円で、アイドル費用がゼロ。個人利用〜小規模なら
月数百円〜数千円で収まる。ローカルGPU購入が正当化されるのは、モデルの
ファインチューニングを頻繁に回すようになった段階。

**オンデバイス完結は不可。** GVHMRクラスの3D復元はスマホでは品質・速度が出ない。
SwingVision等がスマホ内で動かしているのはボール/コート追跡＋軽量2Dであり、
3Dのバイオメカ解析ではない。

「毎回撮って学びにする」で本当に効くのは**履歴の蓄積**。
1回きりの分析はスナップショットでしかない。

| Phase | 内容 | 状態 |
|---|---|---|

| **P1 3Dコア** | 3D復元→指標算出をパイプライン化（現在は手作業） | 次 |
| **P2 動作理解** | サーブ区間の自動切り出し、打点検出（YOLO+音） | |
| **P3 蓄積** | 履歴DB。自分の過去との比較、良い球vs悪い球 | |
| **P4 プロダクト化** | サーバーレスGPU化 + Web UI + 3Dビューア | |
| **P5 プロ比較** | 体格正規化＋DTW＋リファレンス | 最後 |

当初 P3(プロ比較) → P4(フィードバック) の順だったが、**フィードバックは
プロ比較なしで実装できる**と分かったため入れ替えた。プロ比較は最後に回す。

### P0 の結果（2026-07-19）

| 指標 | 値 |
|---|---|
| 重心の高さ範囲 | 0.890 – 1.120 m（身長の約55%＝解剖学的に妥当） |
| 沈み込み | frame 87 / 0.953 m |
| **打点（動画上の実際の接球と一致）** | **frame 103 / 1.120 m** |
| 伸び上がり | +0.167 m / 0.53 秒 |

重心の最高点が実際の打点と一致した。動画のピクセルを見ずに3D重心の波形だけから
打点を特定できたことが、計算が物理的に正しいことの証拠となった。

### 次にやるべきこと

1. **120fpsで再撮影** — キネティックチェーンを評価可能にする
2. **複数サーブでの再現性確認** — 1本では偶然の可能性が残る
3. **サーブ区間の自動切り出し** — 1クリップに他の動作が混在する
4. **人物追跡の頑健化** — 背景に別プレイヤーがいると追跡が移りうる
5. **コート座標の定義** — 現状 水平軸がコートのどちら向きか未定義で
   踏み込み方向を評価できない
