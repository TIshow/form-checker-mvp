# テニスフォーム分析アプリ — ゼロからの再設計 (2026)

## 0. このドキュメントの目的
現行 MVP を捨て、「動画をアップロードすると、自分のフォームを**正確に定量化**し、
**プロと比較して"どこをどう直すべきか"を学びとしてフィードバック**する」アプリを
2026年の最先端手法でゼロから設計するための方針書。

---

## 1. 実現したいこと（ビジョンの整理）

> 動画を撮って「なんか崩れてたね」で終わるのではなく、
> **データに裏付けられた、行動に移せる学び**を返すツール。

具体的に抽出・可視化したい対象：
- **重心 (Center of Mass)** と、その移動・体重移動（前後左右への荷重）
- **キネティックチェーン**（脚→腰→体幹→肩→肘→手首の力の伝達順序と効率）
- **関節角度**（肘の曲がり、肩の外旋、膝の屈曲、股関節など）
- **ヒットポイント**（打点の高さ・体との位置関係・タイミング）
- **サーブ**：トロフィーポーズ、ラケットドロップ、脚のドライブ、体幹の回転、打点
- **プロとの差分**：単なる数値差ではなく「あなたはここが○○だから、△△を直すと良い」

## 2. 現行アプリの根本的な限界（なぜ作り直すべきか）

| 項目 | 現状 | 問題 |
|---|---|---|
| 姿勢推定 | MediaPipe **2D** (33点ピクセル座標) | 奥行きが無い。カメラアングルで角度が歪む。**3Dの体の使い方は測れない** |
| 重心 | 2Dピクセルの質量加重平均 | 物理的に無意味。奥行き・接地・重力方向が無く、**"合ってるか分からない"のは正しい直感** |
| 体重移動 | 無し | 2Dでは前後の荷重が原理的に取れない |
| ボール追跡 | Mock YOLO（ダミー） | 実際には検出していない。ヒットポイントが取れない |
| プロ比較 | フレーム単位の角度差 | 時間同期なし・体格正規化なし・カメラ差補正なし → **差は出るが意味が無い** |
| フィードバック | 赤くハイライトするだけ | 「なぜ・どう直すか」が無い |

→ **2Dベースである限り、目標には原理的に到達できない。3Dへ移行が必須。**

## 3. 2026年の最先端：正しいアプローチ

### レイヤ構成（4段階のパイプライン）

```
動画
 └─▶ ① 3D人体復元 (world-grounded SMPL)   ← ここが現行との決定的な差
     └─▶ ② バイオメカニクス量の算出 (3D関節角・重心・体重移動)
         └─▶ ③ テニス動作の意味づけ (フェーズ分割・キネティックチェーン・打点)
             └─▶ ④ プロとの整列比較 → 行動可能なフィードバック
```

### ① 3D人体復元（コア技術の刷新）
2Dキーポイントではなく、**SMPL**（3D人体メッシュ + 体型）を世界座標で復元する。
2024–2026のSOTA：

- **GVHMR** (SIGGRAPH Asia 2024 / TPAMI 2026) — *推奨*。
  「重力方向」を推定するGravity-View座標系で、**接地・傾き・バランスが安定**して取れる。
  重心と体重移動の分析に本質的に効く。
- **WHAM** (CVPR 2024) — 高速・世界座標での動き復元。実績多い。
- **TRAM** (ECCV 2024) — SLAMベースで軌跡精度は高いが、物理的な破綻が出やすい。

出力：時系列の3D関節位置 + 体型（体節長・質量分布）。
→ これにより**物理的に正しい重心**を3Dで計算できる。

### ② バイオメカニクス量の算出
SMPLの体節に**人体計測学的な質量分布(De Leva等)**を割り当て、
3D世界座標で**全身重心**を計算。さらに：
- 重心の軌跡・速度、支持基底面（両足）に対する重心位置 → **バランス/体重移動**
- 3D関節角度（肘屈曲、肩の内外旋、膝屈曲、股関節、体幹の傾き）
- （発展）**OpenCap / OpenSim** 相当の筋骨格モデルへ変換し、臨床レベルの関節角・トルクへ。
  OpenCapはマーカーレスで妥当性が論文検証済み。

### ③ テニス動作の意味づけ
数値の羅列ではなく「テニスの言葉」に翻訳する層：
- **フェーズ自動分割**
  - サーブ：構え→トス→トロフィー→ラケットドロップ→加速→**打点(contact)**→フォロースルー
  - ストローク：ユニットターン→テイクバック→フォワードスイング→**打点**→フォロースルー
- **キネティックチェーン解析**：各セグメントの角速度ピークが *脚→腰→体幹→肩→肘→手首* の
  順に時間差で連鎖しているか（順序と遅延を定量化 = 力の伝達効率）
- **ヒットポイント検出**：
  - ラケット/ボール検出（**YOLOv8/v11をテニス用にファインチューン**）
  - **打球音（"パコッ"）による接触検出**（音は打点タイミングの強力な信号）
  - 手首・ラケットの速度が最大 → 打点フレーム特定
- **X-factor**（肩-腰の分離角）、打点の高さ・前後位置など指標化

### ④ プロとの整列比較 → フィードバック
naïveなフレーム差分をやめ、正しく比較する：
1. **体格正規化**（体節長で規格化 → 体の大きさの差を除去）
2. **時間整列 (DTW: Dynamic Time Warping)** + フェーズ整合で、
   スイングの速さが違っても「同じ局面同士」を比較
3. **プロのリファレンス・テンプレート**（複数プロ×球種を蓄積）
4. **フィードバックエンジン**：
   - 指標×閾値のルール（コーチング原則に基づく）で逸脱を検出
   - 例：「打点が体の後ろ15cm → 前で捉えるとパワーが上がる」
     「ラケットドロップが浅い → 肩の外旋を深く」
     「重心が後ろ残り → 踏み込みで前方へ荷重を」
   - （発展）指標をLLMに渡し**自然言語コーチング**に変換

## 4. 推奨システム構成

```
[ブラウザ] 動画アップロード / 結果ビューア (Web)
    │  ※3Dモデル推論はGPU必須 → ブラウザ内リアルタイムは不可。サーバ処理。
    ▼
[APIサーバ] FastAPI（ジョブ投入・進捗・結果配信）
    ▼
[推論ワーカー] GPU (PyTorch)
    ├─ 3D人体復元: GVHMR/WHAM
    ├─ ボール/ラケット: YOLO(ファインチューン) + 音声接触検出
    ├─ バイオメカ計算: 重心/関節角/キネティックチェーン
    └─ 比較: DTW + プロテンプレDB
    ▼
[ストレージ] 元動画・3Dモーション(.npz)・指標(JSON)・レンダ動画
    ▼
[3Dビューア] Three.js で骨格/メッシュを自由視点＋指標オーバーレイ
```

- **MVPの近道**：まずは現行のStreamlitのまま、中身を①②③に差し替えて検証。
  UIの作り込み（FastAPI+React+Three.js）は分析ロジックが固まってから。
- **3D推論は重い**（数十秒〜分/動画）。非同期ジョブ前提。

## 5. 段階的ロードマップ

| Phase | 内容 | 得られるもの |
|---|---|---|
| **P0 検証** | GVHMR/WHAMを1本の動画で回し、3D関節+重心が妥当か目視検証 | 技術的実現性の確認 |
| **P1 3Dコア** | 3D復元→3D重心/関節角の算出をパイプライン化 | "正しい"重心・角度 |
| **P2 動作理解** | フェーズ分割＋打点検出（YOLO+音）＋キネティックチェーン | テニスの言葉での指標 |
| **P3 比較** | 体格正規化＋DTW＋プロテンプレ数本 | 意味のある差分 |
| **P4 FB** | ルールベースのフィードバック → LLM自然言語化 | "どう直すか"の学び |
| **P5 プロダクト化** | FastAPI+React+Three.jsビューア、動画レンダ | 使えるWebアプリ |

## 6. 最初の意思決定ポイント（要確認）
1. **対象動作**：サーブ優先か、ストローク優先か、両方か
2. **プロ比較の出典**：プロ動画をどう入手するか（自前撮影/公開映像/API）※権利に注意
3. **撮影条件**：ユーザーの撮影を1カメラ固定前提にできるか（角度の自由度）
4. **MVPの姿勢**：まず分析精度をStreamlitで詰めるか、最初からWebプロダクト設計か
